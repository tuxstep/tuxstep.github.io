TuxSTEP — design document v3.0

A modern NeXT-inspired operating system distribution. Linux kernel underneath, Apple's libsystem and Darwin-derived userland on top, GNUstep frameworks (via Gershwin) for Cocoa-style application development. No glibc, no GNU userland, no Mach-O execution. A clean Darwin-on-Linux architecture.

Tux (Linux kernel) + STEP (NeXTSTEP heritage). v3.0 generated 2026-04-26. The architectural source of truth for the project lives at tuxstep/tuxstep/PLAN.md and supersedes any earlier description in this repo.

Identity

TuxSTEP is a Linux distribution where the userspace is Apple's open-source userland. The Linux kernel provides hardware support and ecosystem reach. From PID 1 upward, every process is libsystem-linked Apple/Darwin-derived code.

LayerSource
Linux kernel + drivers + firmwarekernel.org upstream Linux + linux-firmware
System libcApple's libsystem (apple-oss-distributions/Libc + libplatform + libpthread + libmalloc + libclosure + libdispatch), ported to ELF
Coreutils / shellApple's BSD-derived utilities (file_cmds, shell_cmds, system_cmds, network_cmds, text_cmds) + zsh, all libsystem-linked
Objective-C runtimelibobjc2 (via Gershwin)
Foundation / CoreFoundationlibs-base + libs-corebase (GNUstep, via Gershwin), rebuilt against libsystem
Filesystem layoutNeXT-style: /System, /Network, /Local, /Users

What TuxSTEP is not

The design space here is full of plausible-sounding adjacent projects. TuxSTEP rejects each of them in favor of a sharper identity:

NotBecause
A macOS binary runnerNo mldr, no dyld, no Mach-O execution path. Brew bottles and MacPorts ports do not run unmodified.
A Darling forkDarling targets Mach-O ABI compatibility, which TuxSTEP doesn't pursue. We build Apple's source as ELF instead.
A Mac OS X cloneThe aesthetic is NeXT-inspired, not bug-for-bug Apple-compatible. Where Apple's behavior is bad, we don't preserve it.
"Linux with Apple themes"The userland is genuinely Apple-derived source code, recompiled against libsystem. Themes don't enter into it.
A glibc-based distributionlibsystem is the system libc. glibc isn't present in v0.1 anywhere in the running system.
A GNU/Linux distributionNo GNU coreutils, no bash (zsh), no GCC (clang), no apt. Linux kernel + Apple userland.

Architecture

One coherent stack, ELF-formatted throughout, libsystem-linked end to end.

+-------------------------------------------------------------+
|  Cocoa-style applications (Obj-C, libobjc2, libs-base)      |
+-------------------------------------------------------------+
|  Apple BSD coreutils, zsh, dev tools (clang, lld, ar...)    |
+-------------------------------------------------------------+
|  Foundation (libs-base) | CoreFoundation (libs-corebase)    |
|  libdispatch  |  libobjc2  |  tools-make                    |
+-------------------------------------------------------------+
|        Apple libsystem (libc, pthread, platform, malloc)    |
|        ELF format. /System/Library/Libraries/libSystem.so   |
+-------------------------------------------------------------+
|                    Linux kernel + drivers                   |
|                    KMS, devtmpfs, framebuffer console       |
+-------------------------------------------------------------+

Every userspace binary in TuxSTEP is ELF and links against /System/Library/Libraries/libSystem.so — no exceptions. Every process makes Linux syscalls (translated by libsystem from BSD/Mach syscall conventions to Linux numbers). There is no second runtime, no parallel libc, no chroot, no namespace switching.

The format question. Apple's userland source code can be compiled to either Mach-O or ELF. Mach-O is what macOS runs because XNU loads Mach-O. We're on Linux, so we compile the same source to ELF. The API surface, behavior, and developer experience are identical; the on-disk file format follows the kernel.

Filesystem layout

Gershwin's NeXT-style four-domain layout, with libsystem and tooling installed at standard paths within those domains.

/                                     ISO root
│
├── boot/                              kernel image, initramfs (read by GRUB)
├── lib/
│   ├── modules/<kver>/                kernel modules (mostly empty in v0.1)
│   └── firmware/                      linux-firmware blobs
│
├── System/                            distro-shipped components, read-only
│   ├── Library/
│   │   ├── Libraries/                 libSystem.so, libobjc2.so, libdispatch.so,
│   │   │                              libgnustep-base.so, libgnustep-corebase.so
│   │   ├── Headers/                   Apple + GNUstep headers
│   │   ├── Tools/                     Apple BSD coreutils, zsh, cc, ld, ar, ...
│   │   └── Makefiles/                 gnustep-make
│   └── Applications/                  empty in v0.1
│
├── Network/                           NFS-mounted site content (empty in v0.1)
├── Local/                             per-machine sysadmin installs (empty in v0.1)
├── Users/                             user homes (empty in v0.1)
│
└── dev /proc /sys /tmp /var /etc      kernel-managed runtime + minimal config

No /usr, no /bin, no /sbin. The userland lives at /System/Library/Tools/; libraries at /System/Library/Libraries/. The shell's default $PATH is just /System/Library/Tools.

Future per-domain expansion (post-v0.1) follows the standard NeXT/Gershwin convention: /Local/Users/<name>/ for interactive users, /Network/Library/... for site-wide content, etc. v0.1 ships these as empty directories.

Boot chain (v0.1)

  1. GRUB loads kernel + initramfs from /boot.
  2. Linux kernel boots. KMS drivers built into the kernel come up; devtmpfs auto-mounts via CONFIG_DEVTMPFS_MOUNT=y; framebuffer console renders at native resolution.
  3. initramfs (small, transient) mounts the squashfs as the new root and pivots.
  4. PID 1 = /System/Library/Tools/zsh directly. No init system, no service supervisor, no auth chain. The user is at a # prompt as root on TTY1.

v0.1 is intentionally minimal. launchd, darlingserver, login chain, getty, hot-plug, Mach IPC, and the GUI stack are all post-v0.1 work. The single-user-shell milestone proves the architecture without committing to subsystems whose design isn't yet locked.

Every line in the v0.1 ISO traces back to one of four sources.

Apple-derived source is never modified directly. Each component repository keeps Apple's source as a git submodule and adds a parallel linux-glue/ directory containing only TuxSTEP-authored code. Syncing with Apple's open-source releases is git submodule update --remote.

Phase plan to v0.1

Roughly 10–13 months of focused work for one person. Each phase is bounded; later phases depend on earlier ones.

PhaseComponentEstimate
0tuxstep/libsystem-linux — Apple libsystem ported to ELF; libSystem.so at /System/Library/Libraries/; hello-world compiles and runs4–6 months
1Bootstrap toolchain — clang + lld rebuilt against libsystem; Apple headers at /System/Library/Headers/; self-hosted3–4 months
2Apple BSD coreutils + zsh ported to libsystem; installed at /System/Library/Tools/3–4 months
3Filesystem tooling: mount, umount, fsck.ext4 — minimal subset for ISO2–3 weeks

Phases 0–2 are the long poles. Phases 3–5 fall out quickly once libsystem and the toolchain are stable.

v0.1 demo

The deliverable is a bootable ISO (amd64 + arm64). After GRUB loads the kernel and the squashfs mounts, the user lands at a # prompt at TTY1, no glibc anywhere on the system. From the shell:

# uname -a
Linux tuxstep 6.16.0-tuxstep-amd64 #1 SMP ... x86_64

# ldd /System/Library/Tools/ls
        /System/Library/Libraries/libSystem.so (0x...)

# ls /
System  Local  Network  Users  boot  dev  etc  lib  proc  sys  tmp  var

# ls /System/Library/Libraries/
libdispatch.so       libgnustep-base.so       libobjc2.so       libSystem.so
libgnustep-corebase.so

# cat > /tmp/hello.m <<EOF
#import <Foundation/Foundation.h>
int main(void) {
    @autoreleasepool {
        NSLog(@"Hello from TuxSTEP v0.1");
    }
}
EOF

# /System/Library/Tools/cc -ObjC -lSystem -lobjc2 -ldispatch -lgnustep-base \
        /tmp/hello.m -o /tmp/hello
# /tmp/hello
2026-XX-XX HH:MM:SS.fff Hello from TuxSTEP v0.1

That's the v0.1 milestone. A complete demonstrable Linux kernel + Darwin userland + GNUstep stack on libsystem, end to end, ELF, no glibc.

Staying synced with Apple

The architecture is deliberately designed to make tracking Apple's open-source releases inexpensive and predictable.

  • Apple source is never modified. Patches against apple-oss would create merge conflicts on every upstream release. We avoid this by keeping all Linux-specific code in parallel linux-glue/ directories and treating Apple's source as a read-only submodule.
  • Submodules pinned to specific Apple OSS release tags. Bumping is git submodule update --remote, not a merge.
  • Syscall stubs generated from a table. Adding a new Apple syscall means adding one line to syscall_table.txt; no new hand-written code.
  • Pin to yearly Apple OSS releases, not tip. Apple ships tagged source drops with each macOS major; we track those, not the upstream development branches.
  • CI watches upstream. A weekly build against the latest Apple OSS tag detects breakage early without forcing immediate action.

Estimated ongoing sync work after the initial port: 1–3 days per Apple OSS release.

Deferred to post-v0.1

v0.1 is deliberately small. The following are real and planned, but later:

  • launchd as PID 1 (replaces shell-as-PID-1). Multi-user, service supervision, plist-based service definitions.
  • Mach IPC — libsimple_darlingserver.so as ELF, darlingserver as a system daemon. Enables NSXPCConnection-style IPC and any code path that depends on Mach ports.
  • Login chain — Apple's login, getty, auth subsystem. Multi-user system with /Users/<name>.
  • Hot-plug + modular kernel — tuxstep-uevent, tuxstep-modload; ship more of the kernel as modules; USB devices auto-detect.
  • Networking — DHCP client, configd, wifi support (wpa_supplicant or alternative).
  • Package management — port apple-oss pkgutil + installer.
  • Display server / GUI — libs-gui, libs-back, libs-opal, libs-quartzcore rebuilt against libsystem; X11 client libraries port; eventually Eau as window manager.
  • Workspace, Terminal, demo applications — the actual NeXT desktop experience.

Each is a separately-scoped follow-on milestone after the v0.1 core base is shipping.

Why this shape

Several plausible alternatives were considered and rejected:

AlternativeWhy rejected
Run unmodified Mach-O binaries (Darling-style)Requires maintaining mldr, dyld, binfmt support, and a Mach-O loader stack forever. Buys macOS-binary compatibility, which is not a TuxSTEP goal. The complexity isn't worth it for a project that doesn't ship that use case.
Use glibc with Apple-OSS components as ELF additionsTwo-libc world long-term creates persistent friction (printf from which libc?). Better to commit to one libc — and Apple's libsystem is more stable, BSD-aware, and aesthetically correct for a NeXT-derived project.
Use GNUstep's libobjc2 + Apple's libobjc4 in parallelApple's libobjc4 and libobjc2 maintain separate class registries; can't share an Obj-C runtime. Pick one. libobjc2 is modern, arm64-clean, and free of Apple-ABI constraints.
Substitute Apple's Foundation with GNUstep's at the ABI levelMach-O binaries linked against Apple's Foundation expect specific ivar layouts and method dispatch behaviors. GNUstep's Foundation cannot satisfy them at runtime even via shims. This rules out the Mach-O execution path entirely — and confirms the ELF-only direction.
Build a Darwin personality kernel module to run unmodified libsystem.dylibBrings back the Mach-O loader maintenance burden plus a kernel module across kernel versions. Darling tried this (darling-newlkm) and abandoned it. The userspace darlingserver path is simpler if Mach IPC is needed at all.
Replace glibc with Apple libsystem on a Devuan-derived distributionMixing glibc-linked Devuan packages with libsystem-linked TuxSTEP code in the same process trees is undefined behavior. A clean ISO with no Devuan binaries is the simpler architecture.

The chosen architecture — Linux kernel + libsystem ELF userland + GNUstep stack on libsystem — is the simplest design that satisfies the project's actual goals (NeXT-inspired, Apple-flavored, modern, maintainable, arm64-clean) without inheriting compatibility burdens it doesn't need.
