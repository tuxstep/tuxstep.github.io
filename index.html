TuxSTEP — design document v1.0

A NeXTSTEP-style userspace on a Linux kernel, built from GNUstep + Gershwin + curated Apple OSS sources. Tux (Linux mascot) + STEP (NeXTSTEP heritage). Generated 2026-04-25; reflects the actual Gershwin filesystem layout from ~/gershwin-developer/Library/Sources/tools-make/FilesystemLayouts/gershwin.

Vision

A Linux distribution whose userspace is a direct descendant of NeXTSTEP — built from GNUstep + Gershwin + curated Apple open-source components — with native ELF binaries that boot directly on a Linux kernel. No Darling emulation; no Mach-O wrappers; no half-finished framework stubs. Where Gershwin/GNUstep already provides something good, use it. Where Apple ships open-source code worth keeping, integrate it directly from github.com/apple-oss-distributions at build time, with our patches in our repo.

Filesystem layout (the Gershwin layout, authoritative)

Gershwin's installation domains are defined in ~/gershwin-developer/Library/Sources/tools-make/FilesystemLayouts/gershwin. This is the source of truth — TuxSTEP follows it exactly. The layout is faithful to NeXTSTEP / OpenStep, with three system-wide domains plus a user domain.

The four domains

DomainPurposeRoot pathWho installs here
SystemDistribution-shipped components. Read-only at runtime in production. Includes the OS itself: frameworks, system apps, system tools./System/The distro (TuxSTEP build pipeline)
NetworkNetwork-mounted shared content (NFS, etc.). Site-wide installs that follow the user across machines./Network/Network admin
LocalPer-machine local installs by the system administrator. Roughly analogous to /usr/local in FHS./Local/Local sysadmin (post-install)
UserPer-user installs and preferences. Lives in each user's home directory.~/ (relative to user home)The user themselves

Each domain has the same structural skeleton. Looking up a resource walks: User → Local → Network → System (most-specific wins). This is OpenStep's classic four-domain search path.

Two key things this gets right that my earlier draft did not:
  • The three-domain split is real and at the root of the FS. /System, /Network, /Local are siblings, not nested. This is OpenStep's heritage. Gershwin's tools-make uses these consistently in its install rules.
  • CLI tools live at /System/Library/Tools, not in /usr/bin or /System/Library/CoreServices. Gershwin's defaults, plutil, dscli, gopen, gpbs, say, autogsdoc, etc., all install there via GNUSTEP_SYSTEM_TOOLS. Apple OSS additions we port (defaults, plutil, ifconfig, otool, lipo, etc.) install there too.

The user-home structure deserves its own attention

Each user's home is itself a miniature domain. Inside /Local/Users/jmaloney/:

│       └── <App>.plist                         Per-app preferences (one file per app)
├── Desktop/
├── Documents/
└── Downloads/

This is exactly what defaults read <domain> reads from and defaults write writes to. Compatible with any code that uses NSUserDefaults.

Search-path semantics

When a Cocoa app loads a framework, finds a tool, or reads a preference, GNUstep walks the four domains in order:

  1. User (~/Library/...) — most-specific; user overrides win
  2. Local (/Local/Library/...) — sysadmin overrides
  3. Network (/Network/Library/...) — site policy
  4. System (/System/Library/...) — distro defaults; lowest priority but always present

This is why the layout matters: it's not aesthetic, it's a working dependency-resolution mechanism baked into GNUstep's runtime.

PATH for the shell

Default PATH on a TuxSTEP shell login should reflect the four-domain order, with Linux fallbacks:

PATH=~/Library/Tools:/Local/Library/Tools:/Network/Library/Tools:/System/Library/Tools:/usr/bin:/bin:/usr/sbin:/sbin

So a user's locally-installed defaults wins over the system one, but every domain has its tools available, and Linux-side tools (apt, kernel scripts) are still reachable as fallbacks.

This means:

  • /etc/passwd entries for normal users have HOME=/Local/Users/<name>, not /home/<name>.
  • The dshelper-baked-in user records the agent saw in Gershwin's source already use /Local/Users/<name>/ by default.
  • If you want symlinks for FHS compat (some Linux scripts hardcode /home), add /home → /Local/Users. Document it explicitly.

Mach-on-Linux: deferred to v2 if needed

v1 is pure POSIX/socket. launchd's Mach-bootstrap subsystem becomes dead code on Linux (nothing on a Linux kernel uses Mach to look up service ports, so the bootstrap server has no clients) — patch 002 in ci/patches/launchd/ disables it. The rest of launchd (job spawning, KeepAlive, plist parsing, socket activation) is straight POSIX.

If a future use case demands Mach (e.g. running unmodified Apple binaries that call bootstrap_look_up), revisit then. Three options on the table:

OptionWhatEffort
Pure POSIX (v1)No Mach anywhere; sockets / D-Bus for IPCLow; that's what we ship
Extract darlingserver standalone~7K LOC C++ Mach userspace daemon + ~754K LOC duct-tape XNU stubs as a static lib. Native ELF binaries link a thin libmach.so2-3 months
Hybrid Mach for specific servicesSockets for normal IPC; minimal Mach bootstrap for compat services1.5-2 months

            squashfs-tools xorriso isolinux syslinux-common \
            grub-pc-bin grub-efi-amd64-bin mtools \
            debootstrap ca-certificates xz-utils \
            build-essential git autoconf libtool pkg-config \
            libxml2-dev libssl-dev libffi-dev libgmp-dev libunistring-dev \
            libx11-dev libxext-dev libxmu-dev libxt-dev libxaw7-dev libxrandr-dev libxcursor-dev \
            libfuse-dev libbsd-dev libdbus-1-dev libcairo2-dev libfreetype-dev \
            libkqueue-dev

      - name: Fetch Apple OSS sources
        run: ./ci/build/fetch-apple-sources.sh

      - name: Apply TuxSTEP patches
        run: ./ci/build/apply-patches.sh

      - name: Bootstrap minimal Debian rootfs
        run: |
          # Note: we install Debian packages to /usr/* as normal,
          # then layer Gershwin on top using the gershwin filesystem layout.
          sudo debootstrap --variant=minbase \
            --include=systemd-sysv,init,linux-image-amd64,live-boot,squashfs-tools, \
              ca-certificates,libudev1,libbsd0,libdbus-1-3,libxml2,libcap2,libfuse2, \
              iproute2,isc-dhcp-client,xserver-xorg-core,xinit \
            --components=main \
            bookworm rootfs http://deb.debian.org/debian

      - name: Build & install GNUstep + Gershwin (using the gershwin layout)
        run: |
          sudo git clone https://github.com/gershwin-desktop/gershwin-developer.git rootfs/Developer
          sudo chroot rootfs /bin/bash -c "
            cd /Developer/Library/Sources/tools-make
            ./configure --with-layout=gershwin --prefix=/
            make install
            cd /Developer
            ./Library/Scripts/Bootstrap.sh
            make install GNUSTEP_INSTALLATION_DOMAIN=SYSTEM
          "
          # After this: /System/Library/Frameworks, /System/Library/Tools,
          # /System/Library/Libraries, /System/Applications, etc. are all populated

      - name: Build & install Apple OSS additions
        run: ./ci/build/build-all.sh --rootfs=rootfs

      - name: Replace systemd as PID 1 with launchd-port
        run: |
          sudo install -m 0755 build/launchd rootfs/sbin/init
          sudo cp ci/launchdaemons/*.plist rootfs/System/Library/LaunchDaemons/
          echo "tuxstep" | sudo tee rootfs/etc/hostname

      - name: Set default PATH for shells
        run: |
          # /System/Library/Scripts/PATH.sh sets the four-domain PATH
          sudo cp ci/scripts/PATH.sh rootfs/System/Library/Scripts/
          sudo tee rootfs/etc/profile.d/tuxstep-path.sh >/dev/null <<'EOF'
          # Sourced for all interactive shells
          source /System/Library/Makefiles/GNUstep.sh 2>/dev/null || true
          export PATH="$HOME/Library/Tools:/Local/Library/Tools:/Network/Library/Tools:/System/Library/Tools:$PATH"
          EOF

      - name: Pack squashfs + write GRUB config + build ISO
        run: |
          sudo mkdir -p iso/live iso/boot/grub
          sudo cp rootfs/boot/vmlinuz-* iso/live/vmlinuz
          sudo cp rootfs/boot/initrd.img-* iso/live/initrd
          sudo mksquashfs rootfs iso/live/filesystem.squashfs -comp xz -noappend -e boot
          sudo tee iso/boot/grub/grub.cfg >/dev/null <<'EOF'
          set timeout=1
          set default=0
          menuentry "TuxSTEP" {
            linux  /live/vmlinuz boot=live console=tty0 console=ttyS0,115200n8
            initrd /live/initrd
          }
          EOF
          sudo grub-mkrescue -o tuxstep.iso iso -- -volid TUXSTEP

      - uses: actions/upload-artifact@v4
        with:
          name: tuxstep-iso
          path: tuxstep.iso

Boot sequence

  1. GRUB → loads vmlinuz + initrd
  2. Linux kernel + live-boot initramfs → mounts squashfs as root
  3. launchd as PID 1 → mounts /proc, /sys, /dev, /dev/pts, /dev/shm, /run, /tmp
  4. launchd reads plists from /System/Library/LaunchDaemons/*.plist in dependency order:
    • Networking: com.tuxstep.dhclient.plist
    • System services: com.gershwin.gdomap.plist, com.gershwin.gdnc.plist, com.gershwin.gpbs.plist
    • Auth: com.gershwin.dshelper.plist
    • SSH: com.openssh.sshd.plist (inetd-style)
    • Display: com.tuxstep.x11.plist launches X server on vt7
    • Window manager: com.tuxstep.windowserver.plist (depends on x11)
    • Login: com.gershwin.loginwindow.plist on the X display
  5. LoginWindow waits for credentials. dshelper validates via Gershwin auth.
  6. User logs in → launchd starts user-session LaunchAgents from ~/Library/LaunchAgents/ + /System/Library/LaunchAgents/ → Gershwin workspace (Workspace.app, Dock equivalents) starts.

Phase 1 deliverable / MVP

  1. Boot the live ISO on QEMU/x86_64.
  2. Land on Gershwin LoginWindow on a graphical X display.
  3. Log in as a user (auth via dshelper).
  4. Get a Gershwin workspace with Terminal, TextEdit, SystemPreferences working.
  5. SSH in from another machine on port 22 (Apple's openssh).
  6. From the shell: defaults, plutil, dscli, otool, lipo, ifconfig, route all on PATH from /System/Library/Tools/.
  7. launchd managing system + per-user services via plists.
  8. Filesystem follows the Gershwin layout exactly: /System, /Network, /Local at root; users in /Local/Users/<name>/.

Decisions deferred to post-MVP

  • v2: installable form. Live-ISO works for v1; persistent install (anaconda-style or similar) is v2.
  • v2: package management. apt for Linux-side things; .tspkg for /System/Library/Frameworks and /System/Applications (a tarball + install script invoking make install GNUSTEP_INSTALLATION_DOMAIN=SYSTEM).
  • v2: Mach IPC if specific use cases demand it. Default plan is to never need it.
  • v3: arm64 support. x86_64 only for v1.
  • v3: Apple binary compatibility. If you ever want unmodified Mac binaries, that's the corner-2 port from earlier docs (objc4 + libs-gui-against-objc4 + Mach IPC). Unrelated to TuxSTEP's core mission.
  • License diligence for Debian-archive-eligibility. APSL 2.0's clauses may or may not pass Debian's free-software bar. Defer until v2.
