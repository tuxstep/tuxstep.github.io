TuxSTEP — design document v2.0

A Linux distribution that runs Mach-O binaries natively. Userspace is a hybrid: GNUstep + Gershwin's NeXTSTEP-style desktop running as ELF/glibc, plus an integrated Darling-derived Mach-O runtime that lets brew bottles, MacPorts ports, and third-party .app bundles execute unmodified. Tux (Linux mascot) + STEP (NeXTSTEP heritage). v2.0 generated 2026-04-26; reflects the source registry at github.com/tuxstep/tuxstep/blob/main/ci/apple-sources.yml.

Vision

A Linux distribution where two userspace tracks coexist on the same kernel:

  • ELF / glibc track — the NeXTSTEP-flavored side. Devuan substrate, GNUstep + Gershwin desktop, xlibre, native TuxSTEP applications. Linked against glibc; familiar Linux tooling (gdb, strace, perf) works normally.
  • Mach-O / libsystem-port track — the Mach-capable side. Apple OSS userland (cctools, network_cmds, file_cmds, mDNSResponder, launchd as PID 1) plus user-installed Mac binaries (brew bottles, MacPorts ports, third-party .app bundles). Linked against libSystem.B.dylib via Darling's libsystem-port (forked into the tuxstep org).

The kernel routes binaries to their loader by binary format — both tracks live on the same filesystem, no darling shell-style namespace switch. Where Gershwin / GNUstep already provides something good, use it. Where Apple ships open-source code worth keeping, integrate it directly from apple-oss-distributions, built against libsystem-port. Where Darling has done the hard work of porting Apple's runtime to Linux, fork their components into github.com/tuxstep/darling-* and inherit ten years of upstream effort.

Architecture: dual-track ELF and Mach-O

Two parallel tracks share filesystem and kernel resources, but NOT in-process libraries.

TrackWhat lives therelibc
ELF / glibcDevuan substrate (bash, sshd, apt, dpkg, kernel modules); xlibre; GNUstep (libs-base, libs-gui, libs-back, libs-corebase, libobjc2); Gershwin desktop (Workspace.app, LoginWindow, SystemPreferences, dshelper, gpbs, defaults, plutil, etc.); darlingserver (Mach IPC bridge daemon)/lib/x86_64-linux-gnu/libc.so.6
Mach-O / libsystem-portApple OSS userland (cctools, network_cmds, file_cmds, mDNSResponder, launchd); Darling runtime userspace (dyld, libsystem-port, libpthread, libplatform, libdispatch, security-shim); brew bottles, MacPorts ports, third-party .app bundles/usr/lib/libSystem.B.dylib

The Linux kernel routes binaries by format — ELF kernel loader for ELF, mach-o-binfmt module → /usr/lib/dyld for Mach-O. The tracks coexist transparently — no darling shell ceremony, no chroot, no prefix.

Cross-track rule. A single process loads libraries from one track only. Mach-O processes don't dlopen() ELF .so files; ELF processes don't dlopen() Mach-O dylibs. Mixing them in one address space cross-loads incompatible libcs (heap corruption hazard). Cross-track communication works fine across processes (pipes, sockets, signals, files — libc-agnostic at the kernel boundary).

Consequence: GNUstep's ELF Foundation (libs-base) is unreachable from a Mach-O brew bottle that links Apple's Foundation in-process. Accepted v1 scope cut — most CLI bottles don't link Foundation; pure C/C++ bottles work. Future work could build GNUstep a second time as Mach-O against libsystem-port to cover this case; out of scope for v1.

Filesystem layout

Gershwin's installation domains are defined in ~/gershwin-developer/Library/Sources/tools-make/FilesystemLayouts/gershwin. This is the source of truth — TuxSTEP follows it exactly. Three system-wide domains plus a user domain, faithful to NeXTSTEP / OpenStep.

The four domains

DomainPurposeRoot pathWho installs here
SystemDistribution-shipped components. Read-only at runtime in production. Includes the OS itself: frameworks, system apps, system tools./System/The distro (TuxSTEP build pipeline)
NetworkNetwork-mounted shared content (NFS, etc.). Site-wide installs that follow the user across machines./Network/Network admin
LocalPer-machine local installs by the system administrator. Roughly analogous to /usr/local in FHS./Local/Local sysadmin (post-install)
UserPer-user installs and preferences. Lives in each user's home directory.~/ (relative to user home)The user themselves

Each domain has the same structural skeleton. Looking up a Cocoa-managed resource walks: User → Local → Network → System (most-specific wins). This is OpenStep's classic four-domain search path, baked into GNUstep's runtime.

Search-path semantics

When a Cocoa app loads a framework, finds a tool, or reads a preference, GNUstep walks the four domains in order:

  1. User (~/Library/...) — most-specific; user overrides win
  2. Local (/Local/Library/...) — sysadmin overrides
  3. Network (/Network/Library/...) — site policy
  4. System (/System/Library/...) — distro defaults; lowest priority but always present

The user-home structure

Each user's home is itself a miniature domain. Inside /Local/Users/<name>/:

~/
├── Applications/                                Per-user installed apps
│   └── Admin/
├── Library/
│   ├── Headers/, Libraries/, Tools/, WebApplications/, Documentation/
│   └── Preferences/
│       ├── GNUstep.conf                        Per-user GNUstep config
│       ├── .GNUstepDefaults                    Defaults database
│       └── <App>.plist                         Per-app preferences (one file per app)
├── Desktop/
├── Documents/
└── Downloads/

This is exactly what defaults read <domain> reads from and defaults write writes to. Compatible with any code that uses NSUserDefaults.

PATH for the shell

Default PATH reflects the four-domain order, with Linux fallbacks:

PATH=~/Library/Tools:/Local/Library/Tools:/Network/Library/Tools:/System/Library/Tools:/usr/bin:/bin:/usr/sbin:/sbin

Where users live

User typeHome directory
System accounts (root, sshd privsep, dshelper privsep)/System/Users/<name>/
Regular interactive users/Local/Users/<name>/

/etc/passwd entries for normal users have HOME=/Local/Users/<name>, not /home/<name>. dshelper's user records already use /Local/Users/<name>/ by default. A /home → /Local/Users symlink ships for FHS-compat scripts that hardcode /home.

Why fork the Darling components

Darling has done ~10 years of work pulling Apple's open-source releases (Apple drops sources after each macOS release without active development), patching them to build on Linux (replacing Mach traps with darlingserver IPC, replacing XNU syscalls with Linux syscalls, building without Apple's xcrun), and filling gaps Apple doesn't open-source (libcorecrypto stubs forwarding to OpenSSL, etc.). Reimplementing this is a multi-year project; consuming Darling's work is the practical path.

We fork rather than depend directly because TuxSTEP needs different framework search paths, removal of Apple-private dependencies we don't replicate, patches we don't necessarily upstream, and an independent release cadence aligned with TuxSTEP rather than Darling. Fork pinning matches Darling's monorepo submodule SHAs where applicable — known-good combinations Darling tests with.

security-shim

A TuxSTEP-original Mach-O dylib at github.com/tuxstep/security-shim (not yet created — first integration). Stub Security.framework forwarding SSL APIs (SSLCreateContext, SSLHandshake, SSLRead/Write, SecTrustCreateWithCertificates, SecTrustEvaluate) to OpenSSL. Keychain APIs return errors gracefully; codesign / notarization APIs are no-ops. Lets brew bottles using SecureTransport for TLS work without porting Apple's full Security.framework.

Bootstrap order

Mach-O on Linux faces a chicken-and-egg: Mach-O processes need libsystem-port; libsystem-port needs darlingserver for Mach IPC; darlingserver needs to be running before any Mach-O process exec's. We solve this by making darlingserver an ELF Linux daemon (it has to be — it's the bridge between Mach-O and the Linux kernel) started by initramfs before PID 1.

1. Linux kernel + initramfs starts
2. mach-o-binfmt registers Mach-O as an executable format
   (kernel module OR binfmt_misc + userspace mldr)
3. initramfs starts darlingserver as ELF
4. initramfs pivots to real root, exec()s /sbin/init
5. /sbin/init -> /System/Library/Tools/launchd (Mach-O)
   Kernel routes through mach-o-binfmt -> /usr/lib/dyld
6. dyld loads libsystem-port; libsystem-port connects to
   already-running darlingserver
7. launchd starts as PID 1 with Mach IPC working from instant zero
8. launchd reads plists, starts the rest of the system
   (mDNSResponder, sshd, GNUstep apps, etc.)

No chicken-and-egg because darlingserver is in initramfs, not a launchd-managed plist.

ISO build pipeline

The actual workflow lives at .github/workflows/build.yml in the main repo. Two-stage: builds container images on every push, then builds amd64 + arm64 ISOs inside those containers. needs: dependency between stages avoids any race.

Sketch (current as of v2.0) 
1. Stage 1: build amd64 + arm64 build-environment containers, push to ghcr.
2. Stage 2 (needs Stage 1): for each arch:
   a. debootstrap minimal Devuan rootfs
   b. Install Devuan packages (kernel, X11, etc.)
   c. Clone gershwin-developer; make install GNUSTEP_INSTALLATION_DOMAIN=SYSTEM
   d. Fetch Apple OSS sources (apple-sources.yml registry)
   e. Apply ci/patches/<name>/*.patch
   f. Build Phase 1 (Darling runtime): darling-newlkm, dyld, libsystem-port,
      darlingserver, libpthread, libplatform, libdispatch
   g. Build security-shim
   h. Build Phase 2 (Apple OSS userland) Mach-O against libsystem-port:
      cctools, network_cmds, file_cmds
   i. Build Phase 3 (launchd-managed): launchd, mDNSResponder
   j. Wire darlingserver into initramfs; wire /sbin/init -> launchd symlink
   k. Pack squashfs + GRUB, build hybrid BIOS+UEFI ISO (amd64) or UEFI-only (arm64)
3. Upload to the `continuous` GitHub release tag.

Phases

Phase ordering follows the dependency chain — runtime first, then userland, then system services that depend on the runtime.

PhaseWorkEffort
1Darling runtime layer: darling-newlkm (or binfmt_misc + mldr) for kernel-side Mach-O dispatch; darling-dyld, darling-Libsystem for userspace Mach-O loading; darlingserver as ELF bridge daemon. Build libpthread, libplatform, libdispatch as Apple-source dylibs. Verify Mach-O processes can exec at all by running a minimal test binary.2–3 months
2Apple OSS standalone CLI tools (cctools, network_cmds, file_cmds) built Mach-O against libsystem-port. Each ports cleanly via ci/patches/<name>/. otool -L /System/Library/Tools/launchd and ifconfig on the test ISO are the milestones.1–2 months
3launchd as PID 1 (Mach-O), with mDNSResponder as the first launchd-managed daemon. inittab gone, runlevel concept gone, everything routed through plists.1–2 months
4security-shim for SSL forwarding to OpenSSL. Brew bottles using SecureTransport for TLS work.1–2 months
5GNUstep + Gershwin desktop integration (ELF track) — already mostly works, needs testing on the dual-track system. LoginWindow, Workspace, SystemPreferences come up; X11 (xlibre) starts; user logs in.2–3 months

Effort

~9–14 months for one person, leveraging Darling's existing runtime work. The wide range reflects how much libsystem-port symbol-coverage debugging is needed when running unfamiliar Mac binaries — fundamentally an "issues-tail" problem that resolves into a long tail of small fixes.

What's deliberately deferred

  • Mac binary Foundation — Mach-O bottles linking Apple's Foundation in-process don't load (cross-track rule). Could be added later by building GNUstep a second time as Mach-O against libsystem-port. Out of scope for v1.
  • Apple's full Security.framework port — security-shim's SSL forwarding handles ~70% of brew bottles' TLS needs. Keychain, codesign, and authd-talking utilities don't work. Revisit if specific use cases surface.
  • IOKit — the equivalent on Linux is udev + sysfs + drm. Mac binaries calling IOKit (system_profiler, pmset, ioreg, diskutil, GUI hardware-introspection apps) fail. Narrow benefit for typical brew CLI workloads; months of effort.
  • arm64 / Apple Silicon support — x86_64 only for v1. arm64 has weaker upstream Darling support; defer to v2.
  • Apple binary signature validation — codesign / notarization no-ops. Acceptable for development; not for security-sensitive deployments.
  • v2: installable form — live-ISO works for v1; persistent install (anaconda-style or similar) is v2.
  • v2: package management — apt for ELF/Linux things; .tspkg tarball + install-script for /System/Library/Frameworks and /System/Applications.

Risks

RiskMitigation
darling-newlkm may not be needed at all (last upstream commit 2022; userspace mldr + binfmt_misc might suffice)Verify during Phase 1 spike. The fork exists either way as a starting point.
Darling's darling-dyld and darling-Libsystem are stale (~2.5 years old)Not a problem — Darling itself uses these stale versions because they work. Our fork inherits known limitations, not bleeding-edge bugs.
Bootstrap-order subtleties (initramfs → darlingserver → launchd) may have Linux-distro-specific quirksThe build harness embeds darlingserver and the boot scripts directly so we control the sequence end-to-end.
libsystem-port symbol-coverage gaps surface as "missing symbol" errors when running unfamiliar Mach-O binariesFixed lazily — when a brew bottle fails, add the missing symbol to libsystem-port, repeat. Darling has done most of this work.
Apple OSS tags drift; Darling fork SHAs driftPin tags / SHAs conservatively; refresh patches every ~6 months.
License diligence for Debian-archive eligibilityAPSL 2.0 covers most Apple sources; not in Debian main. Distribute via TuxSTEP's own ISO.
